Item pool for simple types. This item pool is designed to minimize execution time and memory footprint. As a result, some operations are not possible compare to {\ttfamily \doxylink{classph_1_1editor_1_1_t_item_pool}{TItem\+Pool}}, e.\+g., iterating over created items, clear, etc. User is expected to keep track of the handles and use them to iterate the container the way they prefer. When using manual handle management APIs, lost handles cannot be retrieved and those storage slots are effectively leaked (pool dtor will still correctly free the memory though).

\doxysubsection{Detailed Description}
\subsubsection*{template$<$typename Item, CHandle\+Dispatcher Dispatcher = THandle\+Dispatcher$<$\+TWeak\+Handle$<$\+Item$>$$>$$>$\newline
class ph\+::editor\+::\+TTrivial\+Item\+Pool$<$ Item, Dispatcher $>$}
Item pool for simple types. This item pool is designed to minimize execution time and memory footprint. As a result, some operations are not possible compare to {\ttfamily \doxylink{classph_1_1editor_1_1_t_item_pool}{TItem\+Pool}}, e.\+g., iterating over created items, clear, etc. User is expected to keep track of the handles and use them to iterate the container the way they prefer. When using manual handle management APIs, lost handles cannot be retrieved and those storage slots are effectively leaked (pool dtor will still correctly free the memory though). 

Note on the lack of {\ttfamily clear()} method\+: We are not tracking slot validity, so no chance to iterate through valid/constructed items and remove them one by one. Simply clearing the whole pool and return all handles to the dispatcher will face stale handle issues sooner or later, as pre-\/existing valid handles (either the ones in the dispatcher or the ones tracked manually by the user) will suddenly become invalid. A more complex API/policy may solve the problem, but the increased complexity might not worth it, and such feature is not needed currently.


\begin{DoxyTemplParams}{Template Parameters}
{\em Dispatcher} & The dispatcher to use for this pool. Unlike {\ttfamily \doxylink{classph_1_1editor_1_1_t_item_pool}{TItem\+Pool}}, trivial item pool does not require the handle type to be related to {\ttfamily Item} in any way, i.\+e., the item type of handle does not matter. This is useful for using the handle like an opaque pointer. Example\+: Specifying {\ttfamily Item = std\+::string} and {\ttfamily Dispatcher = \doxylink{classph_1_1editor_1_1_t_handle_dispatcher}{THandle\+Dispatcher}\texorpdfstring{$<$}{<}\doxylink{classph_1_1editor_1_1_t_weak_handle}{TWeak\+Handle}\texorpdfstring{$<$}{<}int\texorpdfstring{$>$}{>}\texorpdfstring{$>$}{>}} is valid and {\ttfamily \doxylink{classph_1_1editor_1_1_t_weak_handle}{TWeak\+Handle}\texorpdfstring{$<$}{<}int\texorpdfstring{$>$}{>}} can be used for accessing the pool. \\
